## 基本ルール
- 返答は日本語で行い、箇条書きを使う
- 難しい言葉は短く説明する
- 情報が足りないときだけ質問し、質問は1つずつにする
- テキストファイルは UTF-8 を指定して読む

## 実装ルール
- まず必要最低限でシンプルに作り、機能追加は必要になった時点で行う
- 仕様にないことは勝手に広げない
- 変更は小さく、1回の変更は1目的にする
- 一度に広い範囲をまとめて書き換えない
- 機能追加と整理は分ける
- 設計は SOLID を意識し、依存は外から内へ向ける
- 不要な共通化、早すぎる拡張、大きすぎる基底クラス、何でも入れるヘルパーは避ける

## 可読性ルール
- クラス、メソッド、関数は1つの役割に絞る
- 名前は省略せず、役割が伝わるものにする
- 失敗時、未設定、空データの動きも分かりやすくする
- 重要な処理はテストしやすい形に分け、テスト名は仕様が分かる形にする

## コメントルール
- コメントは処理のまとまりごとに1行書く
- 何をするかを書き、必要なときだけ理由も書く
- クラスやメソッドの上にも必要なコメントを書く
- 既存のコメントはできるだけ消さない

## ドキュメントルール
- 必要に応じてドキュメントを更新する
- 更新順は `SPEC` → `CLASS_DESIGN` → `TASKLIST` → `PROGRESS` → `FAILURE_LOG` とする
- 仕様変更時は `SPEC`、役割や依存変更時は `CLASS_DESIGN` を先に更新する
- 作業完了後は `TASKLIST` `PROGRESS` `FAILURE_LOG` を更新し、完了項目は `[x]` にする
- 実装だけ変えてドキュメントを置き去りにしない

## ドキュメント入口
- 仕様：`./.codex/SPEC.md`
- クラス設計：`./.codex/CLASS_DESIGN.md`
- タスク：`./.codex/TASKLIST.md`
- 進捗：`./.codex/PROGRESS.md`
- 失敗：`./.codex/FAILURE_LOG.md`

## Unityルール
- Inspector参照は最小限にし、起動時の自動探索または注入方式を優先する
- Unity固有の処理はできるだけ外側に置き、中心の処理は Unity API に強く依存させない
- `.unity` `.asset` `.meta` は変更しない
- 変更が必要な場合は事前に依頼する

## 出力ルール
- 変更内容はそのまま使える形で示す
- 実装案はできるだけ小さい変更案から出す
- 変更後は「何を変えたか / なぜ変えたか / 影響範囲 / 読む順番 / 未対応」を短くまとめる
